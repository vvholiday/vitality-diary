# 抗衰老打卡 - 产品设计文档

## 写在前面：先说说免费部署方案的坑

GitHub + Vercel + Supabase 这个组合确实很成熟, 适合 MVP. 但有三个地方需要留意:

### 1. Supabase 免费层限制

- 最大 500MB 数据库, 约能存几十万条打卡记录, MVP 完全够用
- 但免费层不支持备份/导出数据库, 如果数据丢了没法恢复, 建议 MVP 阶段定期手动导出一次
- 免费层每天挂起(如果 7 天没人访问数据库会暂停), 不过 Vercel 有请求时会自动唤醒, 只是第一次访问会慢几秒

### 2. Vercel 免费层限制

- Serverless 函数最长执行 10 秒, 对于打卡这种简单操作完全没问题
- 带宽每月 100GB, 个人用基本用不完
- 最大问题是国内用户访问可能会慢, 因为 Vercel 的服务器默认在海外

### 3. 不能忽略移动端

- 你说明前只要 PC 网站, MVP 确实可以. 但我建议前端用响应式设计, 因为打卡是一个高频碎片化的行为, 用户大概率想在手机上用
- 这个方案我会按 PC 为主手机也能用来设计

## 一、技术架构

### 总览图

用户访问路径：你(或用户)打开网址 -> Vercel 提供网页 -> 网页里操作数据直接读写 Supabase

### 技术选型(简单解释为什么选这些)

| 层 | 选什么 | 为什么 |
|---|---|---|
| 前端框架 | Next.js | 和 Vercel 一家公司的产品, 部署最丝滑; 自带页面路由 |
| 后端 | Next.js API 路由 | 前后端写在一起, 不用单独维护一个后端项目 |
| 数据库 | Supabase(PostgreSQL) | 有免费的托管数据库, 不用自己管服务器 |
| 登录认证 | Supabase Auth | 和数据库是同一家的, 集成方便; 支持邮箱/Google/微信登录 |
| 样式 | Tailwind CSS | 写样式最快最灵活, 做响应式很省事 |
| 部署 | Vercel | 连上 GitHub 后, 你只要把代码推上去, 它自动部署 |

### 关于管理后台和C端的关系

它们不是两个独立网站, 而是一个网站里的两套页面：
- 你登录后看到的是「管理后台」
- 普通用户登录后看到的是「打卡页面」
- 通过登录账号的权限来区分看到什么

这样只部署一次, 省事省钱。

## 二、数据库设计

### 核心思路

用户的数据就是「谁 x 哪天 x 做了哪个项目」这么简单。我按这个思路拆成几张表：

### 表结构

#### 1. users --- 用户表

存登录用户的信息(Supabase Auth 会自动创建这个表, 我们只需要补充自定义字段)

| 字段 | 说明 |
|---|---|
| id | 用户唯一编号(自动生成) |
| email | 邮箱 |
| nickname | 昵称 |
| role | 角色：admin(你)或 user(普通打卡用户) |
| created_at | 注册时间 |

#### 2. categories --- 打卡分类表

你管理后台配置的分类, 比如「运动」、「饮食」

| 字段 | 说明 |
|---|---|
| id | 分类编号 |
| name | 分类名称(如运动) |
| emoji_icon | 小图标, 让页面好看(比如 emoji) |
| sort_order | 排序, 你拖拽调整顺序时用的 |
| is_active | 是否启用(可以暂时隐藏一个分类, 不用删掉) |

#### 3. items --- 打卡项目表

每个分类下的具体项目, 比如「运动」下的「有氧运动」

| 字段 | 说明 |
|---|---|
| id | 项目编号 |
| category_id | 属于哪个分类 |
| name | 项目名称(如有氧运动) |
| unit | 单位, 比如分钟/次/公里(可以不填, 表示只记录了没) |
| description | 可选说明, 比如建议每周 3-5 次, 每次 30 分钟 |
| sort_order | 排序 |
| is_active | 是否启用 |

#### 4. check_ins --- 打卡记录表

核心表, 记录用户每天做了什么

| 字段 | 说明 |
|---|---|
| id | 记录编号 |
| user_id | 谁打的卡 |
| item_id | 打卡了哪个项目 |
| check_date | 哪天(存日期, 不含时间) |
| value | 数值(比如跑了 30 分钟就填 30) |
| note | 备注(用户可写的话, 比如今天状态不错) |
| created_at | 记录创建时间 |

设计上的讲究: 每个用户每天对每个项目只能打一次卡; value 字段让打卡不只是做了/没做的开关, 还能记录量

#### 5. 补充: daily_logs --- 每日总结表(可选)

如果未来想做每日总结/日记功能, 先留个位置

| 字段 | 说明 |
|---|---|
| id | 编号 |
| user_id | 用户 |
| log_date | 日期 |
| mood | 心情评分(1-5星) |
| weight | 体重(可选录入) |
| sleep_hours | 睡眠时长 |
| note | 每日感想 |

## 三、页面结构

### C端(普通用户看到的)

#### 1. 首页 / 登录页

- 简洁的欢迎界面 + 登录/注册入口
- 支持邮箱注册登录, 也支持 Google / 微信一键登录
- 登录后自动跳到打卡页

#### 2. 打卡主页(每天打开看到的核心页面)

设计要点:
- 按分类分组展示打卡项目, 一眼看清今天要做什么
- 每个项目可以填量(分钟/次数等), 也可以只勾选
- 顶部有日历导航, 可以翻到任意一天补打卡
- 底部有完成进度条, 给你成就感

#### 3. 历史记录 / 日历视图

- 月视图日历, 哪天打了卡会标记颜色
- 绿色 = 完成率高, 红色 = 没怎么打
- 点击某天跳转到那天的打卡详情

#### 4. 个人统计

- 周/月/年的打卡完成率折线图
- 连续打卡天数(小成就系统)
- 每个项目的打卡频率排名

#### 5. 个人设置

- 修改昵称、密码
- 退出登录

---

### 管理后台(只有你能看到)

#### 1. 管理后台首页

- 总用户数、今日打卡人数、总打卡次数等关键数据概览

#### 2. 分类管理

- 列表展示所有打卡分类
- 新增/编辑/删除分类
- 拖拽排序

#### 3. 项目管理

- 按分类筛选展示打卡项目
- 新增/编辑/删除项目
- 配置每个项目用什么单位

#### 4. 用户管理(可选)

- 查看注册用户列表
- 查看某个用户的打卡统计

> 安全提醒: 管理后台的入口不能公开暴露, 我会放在一个隐藏路径比如 /dashboard, 只有你的账号(admin 角色)才能访问。

## 四、开发计划

### 整体原则

- MVP 目标是: 你能上线用起来, 自己通过管理后台配项目、自己打卡
- 按独立功能块分阶段, 每个阶段做完都是可用的
- 预估工作日是指我一个人(全栈开发)的实际编码时间

### 第一阶段: 骨架搭建(预估 3-4 天)

| 步骤 | 做什么 | 产出 |
|---|---|---|
| 1 | 初始化 Next.js 项目, 接入 Tailwind CSS | 代码框架搭好 |
| 2 | 在 Supabase 创建数据库表 | 数据结构就绪 |
| 3 | 接入登录注册(邮箱 + Google) | 用户能注册登录了 |
| 4 | 基础布局(导航栏、页面框架) | 所有页面有统一的壳 |

### 第二阶段: 管理后台(预估 2-3 天)

| 步骤 | 做什么 | 产出 |
|---|---|---|
| 1 | 管理后台登录 + 路由保护(只有你能进) | 安全隔离 |
| 2 | 分类管理(增删改查 + 排序) | 你能配分类了 |
| 3 | 项目管理(按分类管理 + 单位配置) | 你能配具体打卡项目了 |
| 4 | 管理后台首页数据概览 | 能看到数据 |

### 第三阶段: 打卡核心功能(预估 3-4 天)

| 步骤 | 做什么 | 产出 |
|---|---|---|
| 1 | 打卡主页: 展示当天全部项目、勾选/填写数值 | 核心打卡流程 |
| 2 | 保存打卡、日期切换、补打卡 | 能自由翻日期 |
| 3 | 每日进度条、完成状态 | 有成就感了 |
| 4 | 历史日历视图 | 能看到打卡轨迹 |

### 第四阶段: 统计与体验(预估 2-3 天)

| 步骤 | 做什么 | 产出 |
|---|---|---|
| 1 | 个人统计页面(完成率图表、趋势) | 有数据洞察 |
| 2 | 连续打卡天数展示 | 小激励 |
| 3 | 响应式适配(手机端也能用) | 手机浏览器也能正常打卡 |
| 4 | 细节打磨(加载状态、空状态、错误提示) | 用起来不毛躁 |

### 第五阶段: 上线(1 天)

| 步骤 | 做什么 |
|---|---|
| 1 | 代码推送到 GitHub |
| 2 | Vercel 关联 GitHub 仓库, 自动部署 |
| 3 | Supabase 连接正式数据库 |
| 4 | 配置自定义域名(可选) |
| 5 | 测试全部流程走通 |

总计: 预估 11-15 个工作日

## 五、提醒你注意的几个点(产品经理视角的避坑指南)

### 打卡的单位要规划好

有些项目只适合做了/没做(比如吃了 5 种颜色的蔬果), 有些适合记录量(跑步 30 分钟). 管理后台配项目时, 我会让你选 这个项目用不用填数值、数值的单位是什么. 如果你的项目五花八门, 建议统一成两种类型:
- 开关型(做了/没做)
- 数值型(填数字 + 单位)

### 补打卡要有道德约束

用户可能会一口气补过去一个月的卡. 要不要限制? 我建议 MVP 阶段允许补打(毕竟有些人刚开始用想记录之前的成果), 但补打的记录在统计时可以标记为补打卡, 不影响连续打卡天数这种成就.

### 数据是你的资产

Supabase 免费层没有自动备份. 建议你每两周做一次数据导出. 我会在后台上加一个一键导出全部数据为 CSV 的功能, 以后你随时可以在 Excel 里分析数据.

### 要不要做社区/社交功能?

MVP 阶段我建议不做. 打卡是件很私人的事, 前期把个人体验打磨好了再说. 以后可以做好友PK、打卡排行榜之类的, 但不是现在.

### 管理后台的权限

这是最容易出安全问题的点. 我建议管理后台入口隐藏在一个不明显的路径(比如 /dashboard 而不是 /admin), 并且每次访问都会检查你的账号权限, 不是管理员就直接跳到打卡页.
